# 语音待机（Voice Standby）设计笔记

> 2026-07 桌面端落地后的观察沉淀，供后续 iOS 移动端设计参考。

## 待机是什么、不是什么

- **是**：一轮转写/代回结束后，保留「目标 App 锁定 + 语音条锚点」一段时间（`voiceStandbySeconds` 配置 0–60），期间再按快捷键跳过重新抓前台窗口，直接对同一会话继续说。
- **不是**：麦克风常开。待机期间不采音，麦只在按下快捷键时开启。UI 上必须让用户能区分这两者（绿点呼吸 ≠ 在听）。
- **Mac 默认 `0`（关）**：热键重进成本低，待机复用易插错窗。代码与本文档保留，供 iOS；桌面要试可在 `~/.fold/config.json` 设 `"voiceStandbySeconds": 8`。

## 核心洞察：待机 = 对「重新进入成本」的补偿

对比三类产品得出的结论（2026-07 调研）：

| 产品/端 | 重新进入成本 | 待机策略 |
|---|---|---|
| Typeless（桌面听写） | ≈0：全局热键常在、跟随光标、不抢焦点、无确认卡 | **不需要待机**，热键本身就是常驻待机 |
| 知更桌面代回 | 中：锁目标窗口 + 截图 + 确认卡 | 短待机（8s），保目标锁定 |
| 豆包 iOS（Seeduplex） | 高：无全局热键、要切 App / 唤起键盘 | 会话级常驻：一次唤醒连续 50+ 轮全双工 |

推论：**成本越高，待机越长**。桌面不要默认常驻——复用旧目标的正确率随时间衰减（用户切窗后往旧窗口插字是事故），且常驻绿点带来"是否一直在听"的信任成本。若埋点（`product_events`）显示大量 15–30s 后重新唤起，再延长。

## 桌面端实现要点（apps/desktop/electron/main.ts）

- 状态：`voiceStandbyUntil / voiceStandbyMode / voiceStandbyPlacement / voiceStandbyActiveApp`。
- 进入：转写插入成功或代回卡片弹出后 `enterVoiceStandby()`；退出：超时 / Esc / agent 模式唤起。
- 空 ASR 走 `fold:voice-empty`、关卡片走 `dismiss({soft:true})`，都只收卡片不退待机。
- **复用规则（Typeless 语义修正）**：待机期间焦点若已切到别的真实 App，放弃复用、跟随当前光标重新锁定。判断必须**同源比对**——进待机时快照 `ContextEngine.activeApp`（System Events 进程名，ignoreApps 已滤掉知更自己），唤起时再取一次比对。
  - ⚠️ 踩坑：不能拿 `voiceTargetApp`（native `localizedName`，如「微信」）和 `ctx.activeApp`（进程名，如 `WeChat`）比，命名不同源必然误判。
  - ⚠️ 踩坑：native `captureTarget()` 会覆盖全局保存的目标输入框，复用路径上不能"试探性抓取"，先用 ContextEngine 判断、确定放弃复用后才抓。
- 复用粒度到 App 为止：同 App 内换输入框仍复用旧 target。

## iOS 版设计约束（做移动端时直接用）

1. **沙盒**：不能向其他 App 注入文字 → 桌面的「目标窗口锁定」概念不存在，替代物是键盘扩展（插入当前输入框）或 App 内会话。
2. **重新进入成本高** → 待机应该更长、甚至会话级常驻：收起键盘不断会话、回到 App 直接继续，参考豆包「一次唤醒持续对话」。
3. **无全局热键** → 唤起入口靠键盘扩展按钮 / 灵动岛 / Action Button / Siri Shortcut，待机的价值比桌面大得多。
4. **后台采音受限**：后台麦克风需常驻音频会话且状态栏会显示红点/胶囊 → "待机不开麦、唤起才采音"的桌面原则在 iOS 同样成立且更刚性。
5. **信任设计**：待机指示必须与录音指示严格区分；iOS 系统本身有麦克风指示灯，UI 不要制造"看起来在听"的歧义。
